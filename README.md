# СТОЛОТО VIP ГОНКИ — Проектная документация

Высокопроизводительная платформа для проведения виртуальных гонок в реальном времени с использованием VIP-баллов, системой Provably Fair и MySQL хранилищем.

## 🌐 Live Demo: (https://website-2k9s.onrender.com/)

Прозрачность: Проект размещен в официальном репозитории, деплой на Render осуществлен через зеркало (https://github.com/Incognit0o0/website) из-за ограничений прав доступа к основному репозиторию.

Примечание по производительности: Приложение развернуто на платформе "Render". В связи с особенностями бесплатного/базового тарифа, при длительном отсутствии активности сервер переходит в режим ожидания («спит»). Первый запрос после пробуждения может занять от 30 до 40 секунд.

##  Доступ
- Администратор: Данные администратора: логин: "admin", пароль: "admin". 
- Пользователи: Регистрация свободна, каждому новому игроку начисляется стартовый баланс 10 000 VIP-баллов.

## 1. Используемый стек технологий

- Frontend: React 18+, Vite, Tailwind CSS 4, Framer Motion (анимации), Radix UI.
- Backend: Node.js, Express, WebSockets (библиотека "ws") для мгновенной синхронизации.
- База данных: MySQL (хранение балансов, истории и параметров комнат).
- Безопасность: SHA-256 хеширование для реализации системы Provably Fair.

## 2. Инструкция по запуску

Приложение готово к работе на Render (https://website-2k9s.onrender.com/). Для локальной разработки:

1.  Сборка проекта:
    ```bash
    npm install
    npm run build
    ```
2.  Настройка БД: Укажите параметры подключения к MySQL в файле ".env" через переменную "DATABASE_URL".
3.  Запуск сервера:
    ```bash
    npm run dev
    ```
    *Сервер автоматически инициализирует таблицы в базе данных при запуске*

## 4. REST API & WebSockets

Приложение использует гибридный подход: REST API для управления состоянием и WebSockets для мгновенной трансляции событий.

### Основные эндпоинты API:

- Авторизация:
    - POST /api/register — Регистрация нового пользователя (первый становится админом).
    - POST /api/login — Авторизация существующего пользователя.
    - GET /api/me — Получение данных текущего сеанса.

- Игровой процесс:
    - GET /api/rooms — Получение списка всех доступных комнат.
    - POST /api/rooms — Создание новой комнаты.
    - POST /api/rooms/:id/join — Участие в заезде (выбор лошадей и списание баллов).
    - POST /api/rooms/:id/boost — Покупка ускорения для гонщика.

- Статистика и администратор:
    - GET /api/history — История последних 50 заездов с деталями Provably Fair.
    - GET /api/admin/stats — Общая статистика платформы (только для админов).
    - POST /api/admin/rooms/update — Изменение параметров комнаты онлайн.

### WebSocket события:

- SUBSCRIBE_ROOM — Подписка клиента на обновления конкретной комнаты.
- ROOM_UPDATE — Рассылка текущего состояния гонки всем подписчикам.
- USER_UPDATE — Уведомление об изменении баланса пользователя.


## 5. Демонстрационные сценарии

### Сценарий 1. Список игровых комнат, фильтрация и рекомендации
Пользователь открывает главный экран и видит список доступных заездов. Вместо прямого поиска используются удобные фильтры по тематикам (Лошади, F1, Космос) и инструменты сортировки по цене входа или количеству мест. Система «рекомендаций» автоматически создает и предлагает новые комнаты с популярными параметрами, когда текущие заполняются, либо отсутствуют.

### Сценарий 2. Вход в комнату, резервирование баллов и буст
При подтверждении участия система списывает сумму ставки с баланса. В лобби игрок может приобрести буст за баллы. Интерфейс мгновенно обновляет вероятность победы, наглядно показывая влияние ускорения на шансы гонщика в предстоящем заезде.

### Сценарий 3. Таймер, заполнение комнаты и real-time запуск
После входа первого игрока запускается таймер, время которого указывает администратор (по дефолту 1 минута). Если к моменту старта остаются свободные места, система автоматически заполняет их ботами. Все изменения — состав участников и тиканье таймера — транслируются всем пользователям в реальном времени через WebSockets.

### Сценарий 4. Определение победителя и визуальный раунд
Backend рассчитывает результат гонки и передает поток координат через сокеты. Пользователь видит плавную анимацию заезда в выбранном оформлении. После финиша система предлагает быстрый переход: повторить участие в той же категории или выбрать более рискованную конфигурацию из списка рекомендованных.

### Сценарий 5. Конфигуратор и оценка параметров комнаты (Admin)
Администратор в реальном времени меняет настройки: цену входа и буста, количество игроков, таймер и комиссию. Система автоматически оценивает прибыльность новой конфигурации для организатора и её привлекательность для игроков, подсвечивая риски при слишком высоких налогах или низких призовых.

### Сценарий 6. Недостаточно бонусных баллов
Если баланса игрока недостаточно для ставки или покупки буста, система блокирует действие и выдает информативное уведомление.

### Сценарий 7. Невалидная конфигурация комнаты (Admin)
При попытке установить противоречивые параметры (например, комиссия 50%+ или отрицательный таймер) конфигуратор блокирует сохранение.

### Сценарий 8. Журнал раундов и прозрачность результата
Любой раунд можно детально разобрать в истории: участники, распределение призового фонда и итоговая прибыль организатора. С помощью Server Seed и SHA-256 хеша пользователь может самостоятельно убедиться, что результат гонки был честным и не менялся в процессе заезда.
